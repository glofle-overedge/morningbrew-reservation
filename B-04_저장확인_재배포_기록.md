# B-04 · 저장 확인·재배포·기록 (Day 04)

- **작성일:** 2026-07-12
- **시나리오:** 시나리오 1 — 카페 「모닝브루」
- **사업명:** 모닝브루 (사장 정해린) — 실습 예시
- **만든 것:** 어제(Day 03) 예약 화면에 localStorage 저장을 적용 (기존 디자인 유지)

## 결과 확인 체크리스트

| 확인 항목 | 이렇게 되면 통과 | 결과 |
|---|---|---|
| 처음 열기 | 저장된 데이터가 있으면 목록에 그대로 보인다 | ✅ localStorage에서 불러와 복원 |
| 새 항목 추가 (Create) | 추가 직후 목록에 보인다 | ✅ setItem 저장 |
| 새로고침 (Read) | 새로고침해도 목록이 유지된다 | ✅ 재열기 후 2건 복원 확인 |
| 삭제 (Delete) | 삭제 후 새로고침해도 삭제 상태가 유지된다 | ✅ 삭제 후 저장함도 1건으로 감소 |
| 전체 초기화 | 모든 항목이 지워지고 빈 상태가 보인다 | ✅ removeItem → 빈 상태 복귀, 버튼 비활성화 |
| Update (선택) | 상태 변경이 새로고침 후에도 남는다 | ✅ 상태(대기중/확인완료/취소) 저장·유지 |

## 산출물 체크

- **B-04-O1** ✅ 예약 추가 후 새로고침해도 목록이 남고, Create·Read·Delete와 전체 초기화가 동작. 데이터는 모두 실습용 가짜 데이터.
- **B-04-O2** ✅ DevTools → Application → Local Storage → `http://localhost:8731`(및 배포 URL)에서 key `reservations`에 JSON 값이 저장된 것 확인. 화면·저장함 어디에도 실제 개인정보 없음(김테스트 / 010-0000-0000 등 가짜 데이터).
- **B-04-O3** ✅ 같은 Vercel 프로젝트에 재배포(과제 취지대로 기존 링크 유지), 배포 URL에서 추가·저장 동작 확인. (localhost와 배포 URL은 origin이 달라 저장소가 분리되는 것도 정상 — 각 주소에서 따로 저장됨.)

## 저장 동작 요약 (내부)

- 추가/삭제/상태변경 → 배열 갱신 → `JSON.stringify`로 포장 → `localStorage.setItem('reservations', ...)`
- 페이지 열 때 → `localStorage.getItem('reservations')` → `JSON.parse`로 풀어 목록 다시 그림
- 삭제 시 바뀐 배열을 다시 저장해야 새로고침 후에도 삭제 유지 (구현 완료)
- 전체 초기화는 `localStorage.removeItem`으로 해당 key만 비움 (`localStorage.clear()` 미사용)
